# Issues Audit Checklist

## Phase 1: Issue Inventory

- [ ] All open issues reviewed
- [ ] Stale issues identified (no activity > 30 days)
- [ ] Duplicate issues flagged
- [ ] Issues without assignees noted
- [ ] Issues without labels noted

## Phase 2: Issue Structure

- [ ] Title is clear and concise
- [ ] Description explains the problem or feature
- [ ] Context provided (what, why, where)
- [ ] Steps to reproduce (for bugs)
- [ ] Expected vs actual behavior (for bugs)

## Phase 3: Objective Clarity

- [ ] Issue states what needs to be done
- [ ] Scope is well-defined
- [ ] API surface specified (api/ or admin/)
- [ ] Success criteria clear
- [ ] Non-goals stated (if relevant)

## Phase 4: Acceptance Criteria

- [ ] Criteria are measurable
- [ ] Criteria are testable
- [ ] Edge cases addressed
- [ ] Error scenarios covered
- [ ] Performance requirements stated (if applicable)

## Phase 5: Scope Boundaries

- [ ] Issue is appropriately sized
- [ ] Large issues broken into sub-issues
- [ ] Dependencies between issues documented
- [ ] Blockers identified
- [ ] Out-of-scope items explicitly noted

## Phase 6: Labels

- [ ] Phase label present (`phase:plan`, `phase:build`, etc.)
- [ ] Type label present (`bug`, `feature`, `enhancement`)
- [ ] Priority indicated (if using priority labels)
- [ ] Surface label present (`api`, `admin`)
- [ ] No contradictory labels

## Phase 7: Duplicates

- [ ] Similar issues cross-referenced
- [ ] True duplicates closed with link to primary
- [ ] Related issues linked
- [ ] Parent/child relationships documented
- [ ] No orphaned sub-issues

## Phase 8: Actionability

- [ ] Issue can be picked up by any qualified agent
- [ ] No critical information missing
- [ ] No blocking questions unanswered
- [ ] Implementation path clear (or marked as needing Plan)
- [ ] No "will figure out later" items
