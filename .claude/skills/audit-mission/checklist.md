# Mission Audit Checklist

## Phase 1: Mission Document

- [ ] README.md exists at root
- [ ] Mission statement present (what this application does)
- [ ] Target users identified
- [ ] Key features listed
- [ ] Quick start path clear

## Phase 2: API Surface Definition

- [ ] Public API surface defined (api/)
- [ ] Admin API surface defined (admin/)
- [ ] Surface purposes documented
- [ ] Consumer differences explained
- [ ] Cross-surface relationships clear

## Phase 3: Purpose Alignment

- [ ] Code structure matches stated purpose
- [ ] Features align with mission
- [ ] No scope creep beyond mission
- [ ] Technical choices support mission
- [ ] Dependencies justified by mission

## Phase 4: Success Criteria

- [ ] Application success defined
- [ ] Key metrics identified (if applicable)
- [ ] User outcomes specified
- [ ] Performance expectations stated
- [ ] Reliability requirements documented

## Phase 5: Non-Goals

- [ ] What this application is NOT stated
- [ ] Out-of-scope features documented
- [ ] Alternative tools referenced (for out-of-scope needs)
- [ ] Boundary with related systems clear
- [ ] Growth constraints acknowledged

## Phase 6: Technical Boundaries

- [ ] Infrastructure requirements documented
- [ ] External dependencies listed
- [ ] Integration points defined
- [ ] Security boundaries established
- [ ] Data ownership clear

## Phase 7: Operational Context

- [ ] Deployment model described
- [ ] Configuration requirements listed
- [ ] Monitoring approach defined
- [ ] Backup/recovery addressed
- [ ] Scaling considerations noted

## Phase 8: Evolution Path

- [ ] Roadmap exists (or backlog)
- [ ] Breaking change policy defined
- [ ] Deprecation approach documented
- [ ] Migration support planned
- [ ] Long-term vision articulated
